## परिचय

यह एक सरल और आसान परियोजना में भाग लेने के लिए पहली बार योगदानकर्ताओं की मदद करने के लिए एक ट्यूटोरियल है।

### उद्देश्य

- ओपन सोर्स प्रोजेक्ट में योगदान देने के लिए।
- Github का उपयोग करने में अधिक सहज होने के लिए।

#### ये किसके लिए है?

- यह बिल्कुल शुरुआती लोगों के लिए है। यदि आप जानते हैं कि एंकर टैग `<a href="" target=""></a>` को कैसे लिखना और संपादित करना है तो आपको यह करने में सक्षम होना चाहिए।
- यह उन लोगों के लिए भी है जिनके पास थोड़ा अधिक अनुभव है लेकिन जो अपना पहला ओपन सोर्स योगदान देना चाहते हैं, या अधिक अनुभव और आत्मविश्वास के लिए अधिक योगदान प्राप्त करना चाहते हैं।

<h5 align="center">मुझे ऐसा करने की आवश्यकता क्यों है?</h5>

किसी भी वेब डेवलपर, आकांक्षी या अनुभवी को Git संस्करण नियंत्रण का उपयोग करने की आवश्यकता है, और GitHub सभी के द्वारा उपयोग की जाने वाली सबसे लोकप्रिय Git होस्टिंग सेवा है। यह ओपन सोर्स कम्युनिटी का दिल भी है। GitHub का उपयोग करके सहज होना एक आवश्यक कौशल है। किसी प्रोजेक्ट में योगदान देने से आपका आत्मविश्वास बढ़ता है और आपको अपने GitHub प्रोफ़ाइल पर दिखाने के लिए कुछ मिलता है। <br>
यदि आप एक नए डेवलपर हैं और आप सोच रहे हैं कि क्या आपको Git और GitHub सीखने की आवश्यकता है, तो यहां उत्तर है: [आपको कल Git सीखना चाहिए था।](https://codeburst.io/number-one-piece-of-advice-for-new-developers-ddd08abc8bfa 'नए डेवलपर? आपको कल Git सीख लेनी चाहिए थी। CodeBurst.io के निर्माता ब्रैंडन मोरेली द्वारा').

<h4 align="center">मैं क्या योगदान करने जा रहा हूँ?</h4>

![Contributor Card](/readme-only/card.PNG 'योगदानकर्ता कार्ड')

आप इस [प्रोजेक्ट के वेब पेज](https://syknapse.github.io/Contribute-To-This-Project/ 'https://syknapse.github.io/Contribute-To-This-Project') पर ठीक इसी तरह एक कार्ड का योगदान करने जा रहे हैं। इसमें आपका नाम, Twitter हैंडल, एक संक्षिप्त विवरण और आपके द्वारा अनुशंसित Web Developers के लिए उपयोगी संसाधनों के 3 लिंक शामिल होंगे।

आप HTML फ़ाइल के अंदर कार्ड टेम्प्लेट की एक प्रति (Copy) बनाएंगे और इसे अपनी जानकारी के साथ अनुकूलित (Customize) करेंगे।

<h4 align="center">Setup करना! :)</h4>

पहले, आइए काम करने के लिए Setup करें

1. अपने Github खाते में लॉग इन करें। अगर आपके पास अभी तक कोई खाता नहीं है तो [GitHub से जुड़ें।](https://github.com/join) मैं अनुशंसा करता हूं कि आप जारी रखने से पहले [GitHub Hello World tutorial](https://guides.github.com/activities/hello-world/) करें।
1. [GitHub Desktop app](https://desktop.github.com/) डाउनलोड करें
   - वैकल्पिक रूप से यदि आप कमांड लाइन पर गिट का उपयोग करने में सहज हैं तो आप ऐसा कर सकते हैं यहां [CLI tutorial](terminal-tutorial.md) का लिंक दिया गया है।
   - यदि आप [VS Code](https://code.visualstudio.com/ 'Visual Studio Code website') का उपयोग करते हैं तो यह एकीकृत (Integrated) Git के साथ आता है और आपको वह करने की अनुमति देता है जो हमें सीधे संपादक (Editor) से चाहिए।
   - हालाँकि इस Tutorial का अनुसरण करने का सबसे सरल और आसान तरीका GitHub डेस्कटॉप का उपयोग करना है।

<h5 align="center">चरण 3: एक नई Branch बनाएँ</h5>

- एक बार Repo Clone और आपने इसे GitHub डेस्कटॉप में खोलने के बाद,
  यहाँ एक नई शाखा बनाने का समय है।
- एक Branch आपके परिवर्तनों को Project के मुख्य भाग से अलग रखने का एक तरीका है जिसे `Master` कहा जाता है। उदाहरण के लिए यदि चीजें गलत हो जाती हैं और आप अपने बदलावों से खुश नहीं हैं तो आप केवल Branch को हटा सकते हैं और Main Project प्रभावित नहीं होगी।

| <ul><li>_Current branch_ पर क्लिक करें।</li><li>इसके बाद _New_ पर क्लिक करें। </li></ul> | ![Create branch](/readme-only/branch-new.PNG "'Branch', इसके बाद 'New' पर क्लिक करें। ") |
| :--------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------: |


| <ul><li>अपनी Branch को एक नाम दें।</li><li>`Create branch` पर क्लिक करें। </li></ul> | ![Name branch](/readme-only/branch-name.PNG 'अपनी Branch को एक नाम दें।') |

**`Master` branch पर काम न करें।**

<h5 align="center">चरण 7: अपने परिवर्तनों को प्रतिबद्ध (Commit) करें</h5>

- GitHub डेस्कटॉप ऐप पर वापस जाएं।
- आपके परिवर्तन स्वचालित रूप से स्टेजिंग क्षेत्र में जोड़ दिए जाएंगे।
- इसका अर्थ है कि Git ने सभी **सहेजे गए** परिवर्तनों को रिकॉर्ड कर लिया है।
- आप इसे ऐप में परिलक्षित देख सकते हैं। आपने जो कुछ भी फ़ाइल में जोड़ा है वह हरे रंग में होगा, और विलोपन() लाल रंग में दिखाई देगा।

<h5 align="center">चरण 8: अपने परिवर्तनों को GitHub पर Push करें।</h5>

- आपके परिवर्तन अब सहेजे गए हैं या Commited हैं। लेकिन वे केवल स्थानीय रूप से सहेजे जाते हैं, यानी आपके कंप्यूटर पर।
- GitHub पर अपने रिपॉजिटरी के साथ स्थानीय परिवर्तनों को Synchronize करना _Push_ कहलाता है । आप अपने स्थानीय रिपॉजिटरी से GitHub पर दूरस्थ (Remote) रिपॉजिटरी में परिवर्तन को "Push" कर रहे हैं।

<h5 align="center">चरण 9: एक PR जमा करें (Pull Request)</h5>

- यह वह क्षण है जिसका आप इंतजार कर रहे हैं; एक _Pull Request_ (PR) जमा करना।
- अब तक आपने जो भी काम किया है, वह प्रोजेक्ट के Fork पर किया गया है, जो कि आपको याद है कि आपके अपने Github खाते में रहता है।
- अब समय आ गया है कि आप अपने परिवर्तनों को Main Project में Merge के लिए भेजें।
- इसे एक [_Pull Request_](https://help.github.com/articles/about-pull-requests/ 'Pull requests के बारे में - GitHub मदद') कहा जाता है क्योंकि आप मूल प्रोजेक्ट अनुरक्षक (maintainer) को अपने प्रोजेक्ट में अपने परिवर्तनों को "खींचने" के लिए कह रहे हैं।
- GitHub पर अपने **Fork** के मुख्य पृष्ठ पर जाएं (इसमें शीर्ष पर Fork आइकन और आपका अपना उपयोगकर्ता नाम होगा)।
- रेपो के शीर्ष की ओर आपको हरे बटन के साथ Highlight किया गया Pull Request संदेश दिखाई देगा।

- सभी विकल्पों से घबराएं नहीं। अभी के लिए आपको केवल ये तीन चरण करने होंगे।
- `Allow edits from maintainers` टिक किए हुए विकल्प को छोड़ दें ।
- अब, प्रोजेक्ट मेंटेनर को एक _Pull Request_ भेजा जाएगा। जैसे ही इसकी समीक्षा की जाती है और इसे स्वीकार कर लिया जाता है, आपके परिवर्तन [प्रोजेक्ट वेब पेज](https://syknapse.github.io/Contribute-To-This-Project 'Contribute To This Project web page') पर दिखाई देने लगेंगे ।

<h5 align="center">चरण 10: जश्न मनाएं !!</h5>

यही बात है। आपने इसे कर लिया है। आपने अब GitHub पर ओपन सोर्स में योगदान दिया है।

आपने लाइव वेब पेज में Code जोड़ा है: [https://syknapse.github.io/Contribute-To-This-Project](https://syknapse.github.io/Contribute-To-This-Project)

आपके परिवर्तन **तुरंत दिखाई नहीं देंगे** ; पहले उन्हें प्रोजेक्ट Maintainer द्वारा समीक्षा, स्वीकार और Merge करना होगा। एक बार उनका Merge हो जाने के बाद आपका कार्ड दिखाई देना चाहिए और पृष्ठ पर लाइव होना चाहिए।

एक Reviewer के लिए PR में बदलाव के लिए पूछना बहुत सामान्य है। अगर आपके साथ ऐसा होता है तो इसे अच्छा अभ्यास समझें। टिप्पणियों और अनुरोधित परिवर्तनों पर नज़र रखें। एक बार जब आप अनुरोधित परिवर्तन (अपनी शाखा में वापस) कर लेते हैं, तो आपको केवल अपने परिवर्तनों को Commit और Push करना होता है। नए बदलावों के साथ PR अपने आप अपडेट हो जाएगा।

मैं वादा करता हूं कि मैं जितनी जल्दी हो सके समीक्षा और विलय करने की कोशिश करूंगा लेकिन मैं इसे अपने खाली समय में करता हूं, इसलिए कुछ दिनों की देरी अपरिहार्य (Inevitable) है।
